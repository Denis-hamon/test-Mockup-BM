    **1. Create template server actions** at `apps/web/src/server/actions/template.actions.ts`:

    All actions use `"use server"` directive. Import requireAvocat from auth pattern used in existing lawyer-settings.actions.ts. Import db, intakeTemplates, intakeTemplateSnapshots from schema.

    - `getSpecialtyTemplates()`: Returns the 3 seed templates (import from seed file) as an array with specialty, label, questionCount. No auth required (public for template selection display).
    - `getTemplateForLawyer()`: requireAvocat() -> query intakeTemplates where lawyerId = session.user.id and isActive = 1. Return the template or null. Single active template per lawyer for v1.
    - `getTemplateBySlug(slug: string)`: Public action (no auth). Query intakeTemplates where slug = slug and isActive = 1. Return template with schema, branding fields, or null if not found. Used by /intake/[slug] page.
    - `saveTemplate(data: { specialty: string, schema: IntakeTemplate, logoUrl?: string, accentColor?: string, welcomeText?: string, slug?: string })`: requireAvocat(). Validate schema with intakeTemplateSchema.parse(). If lawyer already has a template, UPDATE it. Otherwise INSERT. Handle slug unique constraint violation gracefully (catch Postgres unique violation error code 23505, return { success: false, error: "slug_taken" }). Return { success: true, id: string }.
    - `checkSlugAvailability(slug: string)`: requireAvocat(). Query intakeTemplates where slug = slug AND lawyerId != current user. Return { available: boolean }.
    - `createTemplateSnapshot(templateId: string)`: Internal helper (not exported as server action). Insert current template schema into intakeTemplateSnapshots, return snapshot ID. Called during intake submission.

    **2. Update submitIntake** in `apps/web/src/server/actions/intake.actions.ts` per D-03:

    Add optional `templateId` parameter to submitIntake. When templateId is provided:
    - Fetch the template, create a snapshot via createTemplateSnapshot
    - Include templateId and templateSnapshotId in the insert values
    - Store the dynamic form answers in a new `templateAnswers` field (add to intakeSubmissions as text column storing JSON) alongside the existing fixed fields

    **3. Create DynamicField component** at `apps/web/src/components/intake/dynamic-field.tsx`:

    "use client" component. Props: { question: TemplateQuestion, form: UseFormReturn<Record<string, unknown>>, visible: boolean }.
    - If !visible, return null
    - Render Label with question.label, required asterisk if question.required
    - Render question.description as muted text if present
    - Switch on question.fieldType to render: Input (text), Textarea rows=3 (textarea), Select with SelectContent/SelectItem from options (select), Input type="date" (date), Checkbox (checkbox), Input type="number" (number)
    - Use form.register(question.id) for text/textarea/number/date. Use form.setValue for select/checkbox (controlled).
    - Show error message from form.formState.errors[question.id] with aria-describedby
    - Follow existing component patterns: shadcn/ui imports, cn() utility, Lucide icons

    **4. Create DynamicStep component** at `apps/web/src/components/intake/dynamic-step.tsx`:

    "use client" component. Props: { step: TemplateStep, allQuestions: TemplateQuestion[], form: UseFormReturn<Record<string, unknown>> }.
    - Import isQuestionVisible from @legalconnect/shared
    - For each question in step.questions, compute visibility using isQuestionVisible(question, allQuestions, form.watch())
    - Render DynamicField for each question with computed visibility
    - Wrap in a div with space-y-4 for field spacing
    - Use form.watch() to reactively update visibility when form values change
