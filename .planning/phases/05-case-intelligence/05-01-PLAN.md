**CASE_SUMMARY_OVERLAY** — System prompt for generating fiche synthetique:
```
Vous etes un assistant juridique specialise dans la synthese de dossiers.
Votre role est de produire une fiche synthetique structuree a partir des informations d'intake et des documents fournis.

## Structure de sortie (JSON strict)
Retournez un JSON avec les champs suivants :
- clientIdentity: string (resume de l'identite du client: nom, contact)
- problemType: string (categorie juridique: droit du travail, famille, penal, immobilier, affaires, autre)
- situationSummary: string (resume narratif de 3-5 phrases, factuel, sans jugement juridique)
- keyFacts: string[] (faits cles extraits des documents et du recit, max 10)
- opposingParties: string[] (noms et descriptions des parties adverses)
- urgencyAssessment: string (evaluation factuelle de l'urgence: delais legaux, risques immediats)

## Regles
- Soyez factuel et objectif. Ne qualifiez JAMAIS juridiquement la situation.
- Synthetisez les informations du formulaire ET des documents uploades.
- Si des informations contradictoires existent entre le recit et les documents, signalez-le.
- Les faits cles doivent etre dates quand possible.
```

**TIMELINE_EXTRACTION_OVERLAY** — System prompt for chronological timeline:
```
Vous etes un assistant specialise dans l'extraction de chronologies a partir de dossiers juridiques.
Construisez une timeline des evenements a partir du recit client et des documents fournis.

## Structure de sortie (JSON strict)
Retournez un JSON avec :
- events: Array<{ date: string, description: string, source: "client" | "document", confidence: "high" | "low", documentId?: string }>
- undatedEvents: Array<{ description: string, source: "client" | "document", documentId?: string }>

## Regles sur les dates
- Dates certaines (explicites dans document) : confidence "high", format "YYYY-MM-DD" ou "YYYY-MM"
- Dates incertaines (deduites du contexte) : confidence "low", prefixer avec "~" (ex: "~2024-03")
- Evenements sans date : placer dans undatedEvents
- Ordonner les events par date croissante

## Regles generales
- Chaque evenement doit avoir une description factuelle courte (1-2 phrases max)
- Indiquer la source : "client" si issu du recit, "document" si extrait d'un document
- Si un document est la source, inclure le documentId
- Ne pas interpreter juridiquement les evenements
```

**QUALIFICATION_SCORING_OVERLAY** — System prompt for scoring:
```
Vous etes un assistant d'evaluation de dossiers juridiques.
Evaluez le dossier selon trois criteres et produisez un score de 0 a 100.

## Structure de sortie (JSON strict)
Retournez un JSON avec :
- urgencyScore: number (0-100)
- completenessScore: number (0-100)
- complexityScore: number (0-100)
- rationale: string (explication en 2-3 phrases de l'evaluation)

## Criteres d'evaluation

### Urgence (0-100)
- 80-100 : Delai legal imminent (<7j), mention de danger, procedure en cours
- 50-79 : Delai dans les semaines, mots-cles urgents, conflit actif
- 20-49 : Situation stable mais necessitant attention
- 0-19 : Demande de renseignement, pas d'urgence apparente

### Completude (0-100)
- Points pour : description detaillee (+30), documents joints (+20 par doc, max +40), coordonnees completes (+10), partie adverse identifiee (+10), dates precises (+10)
- Deduire pour : champs vides, contradictions, informations vagues

### Complexite (0-100)
- Facteurs : nombre de parties adverses, domaine juridique (penal/famille = +20), montants en jeu, nombre de documents, duree du litige

## Regles
- Soyez objectif et reproductible dans votre evaluation
- Le rationale doit justifier les scores en termes factuels
- Ne donnez JAMAIS de conseil juridique dans le rationale
```

**1. Create `apps/web/src/server/services/case-intelligence.service.ts`**

This service contains three AI generation functions. Use `generateText` from Vercel AI SDK (NOT streaming — these are background jobs, no UI to stream to). Import `getModel` from `@legalconnect/ai`, import the overlay constants.

```typescript
import { generateText } from "ai";
import { getModel, buildSystemPrompt, CASE_SUMMARY_OVERLAY, TIMELINE_EXTRACTION_OVERLAY, QUALIFICATION_SCORING_OVERLAY } from "@legalconnect/ai";
```

**`generateCaseSummary(input: CaseIntelligenceInput): Promise<CaseSummaryResult>`**
- Build user message assembling: intake submission data (problemType, description, urgency, opposingParty, desiredOutcome, fullName), AI follow-up Q&A pairs, and extraction results (dates, parties, amounts, keyClauses, summary per document).
- Call `generateText({ model: getModel(), system: buildSystemPrompt(CASE_SUMMARY_OVERLAY), messages: [{ role: "user", content: userMessage }], maxTokens: 4096 })`.
- Parse the JSON response. Return typed `CaseSummaryResult` with fields matching the caseSummaries table columns.
- Wrap JSON parsing in try/catch — if AI returns malformed JSON, throw with "INVALID_AI_RESPONSE" prefix.

**`generateTimeline(input: CaseIntelligenceInput): Promise<TimelineResult>`**
- Same input assembly as summary but emphasize dates and temporal references.
- Use TIMELINE_EXTRACTION_OVERLAY. maxTokens: 4096.
- Parse JSON. Return `TimelineResult` with events array and undatedEvents array.
- Sort events by date ascending after parsing.

**`generateQualificationScore(input: CaseIntelligenceInput): Promise<QualificationScoreResult>`**
- Same input assembly.
- Use QUALIFICATION_SCORING_OVERLAY. maxTokens: 2048.
- Parse JSON. Compute `overallScore = Math.round((urgencyScore * 40 + completenessScore * 35 + complexityScore * 25) / 100)`.
- Clamp all scores to 0-100 range.
- Return `QualificationScoreResult`.

**2. Create `apps/web/src/server/workers/case-intelligence.worker.ts`**

Follow the exact pattern from extraction.worker.ts (per Phase 4 decision: BullMQ worker as separate process).

```typescript
export interface CaseIntelligenceJobData {
  submissionId: string;
  version: number; // per D-07: track version for regeneration
}
```

- Create queue: `new Queue<CaseIntelligenceJobData>("case-intelligence", { connection })`
- Export as `caseIntelligenceQueue`.

Worker logic:
1. Mark all three tables' rows as "processing" (the pending rows were created by the trigger action in Plan 02).
2. Gather all data for this submission:
   - Query intakeSubmissions by submissionId
   - Query aiFollowUps where submissionId matches and answer is not null and skipped = 0
   - Query intakeDocuments + extractionResults (join) where submissionId matches and extraction status = "done"
3. Build `CaseIntelligenceInput` from gathered data.
4. Run all three generation functions IN SEQUENCE (not parallel — each benefits from previous context, and parallel would double API cost for no gain since they share the same input assembly):
   - `generateCaseSummary(input)` -> update caseSummaries row
   - `generateTimeline(input)` -> update caseTimelines row
   - `generateQualificationScore(input)` -> update qualificationScores row
5. On individual failure: mark that specific table's row as "failed" with error, continue to next. The other artifacts should still be generated even if one fails.
6. Return summary of results `{ summary: "done"|"failed", timeline: "done"|"failed", score: "done"|"failed" }`.

Worker config: `concurrency: 2` (heavier than extraction — each job makes 3 LLM calls).

Add event handlers (completed, failed) following extraction.worker.ts pattern.
